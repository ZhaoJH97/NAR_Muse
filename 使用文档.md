# Muse-NAR：人声转伴奏（V2A）非自回归模型 · 中文使用文档

本项目在 [Muse](https://github.com/yuhui1038/Muse) 的 `qwen3-0.6B-music` 基模之上，
实现了一个**非自回归（NAR）的人声转伴奏（Vocal-to-Accompaniment, V2A）模型**：
输入「人声 codec token + 被掩码的伴奏 codec token」，输出「完整的伴奏 codec token 序列」。
方法参考论文 **LaDA-Band**（[arXiv:2604.11052](https://arxiv.org/pdf/2604.11052)），
把它的 LLaMA-3.2-1B 骨干换成了 Muse 的 Qwen3-0.6B 音乐骨干。

> 一句话概括：把「文本→音乐的自回归 LLM」改造成「人声→伴奏的双向掩码扩散模型」，
> 复用基模的 Transformer 层和 `<AUDIO_*>` 音频 token 嵌入。

## 一、整体设计原理

### 1.1 任务定义
给定一段人声（vocal）的离散 codec token 序列 `V = v_1..v_T`，生成与之**逐帧对齐**的
伴奏（accompaniment / BGM）token 序列 `A = a_1..a_T`。两者都来自同一个 MuCodec
码本（25 Hz、单码本、码本大小 16384），所以长度一致。

### 1.2 为什么用非自回归 + 掩码扩散
- **自回归（AR）**：逐 token 生成，长序列会误差累积、编排漂移，且显存/算力随长度暴涨。
- **本项目（NAR 掩码扩散）**：一次性预测所有被掩码位置，**双向注意力**能同时看到整段人声和
  整段（部分可见的）伴奏，再通过多步迭代「由粗到细」地精修。长程一致性和与人声的同步更好。

### 1.3 模型数据流（核心）
```
 人声 token   v_1..v_T  ──[vocal_embed]──┐
                                         ├─ 逐帧特征维拼接 ─[Linear: 2E→D]→ X ∈ R^{T×D}
 掩码伴奏 ã_1..ã_T      ──[acc_embed]────┘
                                         │
                                         ▼
                  ┌────────────────────────────────────────┐
                  │   Qwen3-0.6B 骨干（双向注意力，去因果掩码） │
                  └────────────────────────────────────────┘
                                         │
                    hidden ∈ R^{T×D} ──[acc_head]→ logits ∈ R^{T×16384}
```
这正是你最初要求的「人声 codec token 和 mask 后的 bgm codec token 拼接（逐帧 embedding
concat 后 linear 到 LLM 的 embedding size）」。

### 1.4 训练目标
- **主损失 CML（条件掩码建模，论文 Eq.3）**：每个伴奏 token 以概率 `t` 独立替换成
  `[MASK]`，模型重建原始 token，损失按 `1/t` 加权（语言扩散模型的标准做法）。
- **辅助损失 RTD（替换 token 检测，论文 Eq.4，ELECTRA 风格）**：让模型判断每个伴奏
  token 是「原始的」还是「被替换的」，给全序列提供稠密监督，缓解 intro/间奏等人声稀疏区
  伴奏过空的问题。总损失 `L = L_CML + λ·L_RTD`（λ=0.2）。

### 1.5 推理
从「全掩码」开始，按**余弦调度**迭代 `num_steps` 步（默认 20）：每步预测全部掩码位置，
保留置信度高的预测，把置信度最低的一部分重新掩码进入下一步（low-confidence remasking）。

## 三、每个模块的实现逻辑

### 3.1 `lada_band/tokens.py` —— token id 约定与 .pt 读取
定义全局常量（与论文 LaDA-Band 5.2 节一致）：

| 常量 | 取值 | 含义 |
|------|------|------|
| `NUM_AUDIO_TOKENS` | 16384 | MuCodec 码本大小 |
| `CODEC_FRAME_RATE` | 25 | 帧率（Hz） |
| `ACC_MASK_ID` | 16384 | 伴奏轨的 `[MASK]` token |
| `ACC_PAD_ID` | 16385 | 伴奏轨 padding |
| `VOCAL_PAD_ID` | 16384 | 人声轨 padding（人声不需要掩码） |
| `ACC_INPUT_VOCAB` | 16386 | 伴奏嵌入表大小 = 16384 + mask + pad |
| `VOCAL_INPUT_VOCAB` | 16385 | 人声嵌入表大小 = 16384 + pad |
| `ACC_OUTPUT_VOCAB` | 16384 | 预测头类别数（mask/pad 永远不是预测目标）|

- `load_codec_pt(path)`：读取 MuCodec 的 `.pt`（`encode_audio.py` 输出形状 `[1,1,T]`），
  鲁棒地兼容 `[T]/[1,T]/[1,1,T]`，压扁成一维 `LongTensor`，并校验所有 id 落在 `[0,16383]`。

### 3.2 `lada_band/config.py` —— 配置
三个 dataclass：`ModelConfig` / `DataConfig` / `TrainConfig`，由 `Config.from_yaml()`
从 YAML 加载，`save_yaml()` 落盘（每个 checkpoint 都会存一份 `config.yaml` 供推理复现）。
详见[第六节](#六配置参数详解)。

### 3.3 `lada_band/masking.py` —— 掩码调度与前向掩码
- `cosine_mask_prob(r)`：把进度变量 `r∈[0,1]` 经余弦调度映射到掩码概率 `t=cos(π/2·r)`
  （`r=0→t=1` 全掩码，`r=1→t=0` 不掩码）。
- `SobolMaskSampler`：用**Sobol 低差异序列**采样 `r`（比均匀随机更均匀地覆盖各种掩码比例），
  再转成每个样本的掩码概率 `t`。
- `forward_mask(acc_ids, pad_mask, t, mask_id)`：对每个伴奏 token 以概率 `t` 独立写成
  `[MASK]`；并**保证每个样本至少掩码 1 个 token**（否则 CML 的 `1/t` 加权会除零）。
  返回 `masked_acc`（送入模型的输入）和 `mask_positions`（布尔，标记哪些被掩了，用于算损失）。

### 3.4 `lada_band/sampling.py` —— 采样工具
- `_top_k_top_p_filter`：标准的 top-k + 核采样（top-p）logits 过滤，保证至少保留 top-1。
- `sample_tokens(logits, temperature, top_k, top_p)`：每个位置采样一个 token，并返回该
  token 在**未过滤** softmax 下的概率作为「置信度」（用于推理时的重掩码决策）。
- `gumbel_noise_like(x)`：采样 Gumbel(0,1) 噪声，用于推理时给置信度加扰动（mask 温度）。

### 3.5 `lada_band/model.py` —— 模型主体（最核心）
类 `LaDABandV2A`，由以下部分组成：

**(1) 骨干构建 `_build_backbone(cfg)`**
- 若给了 `pretrained_backbone`：用 `AutoModelForCausalLM.from_pretrained` 载入你的基模，
  取其 `.model`（即 `Qwen3Model`，含 28 层 + RMSNorm + RoPE）作为骨干。
- **Warm-start（关键）**：当 `track_embed_dim == hidden_size` 时，用基模 tokenizer 把
  `<AUDIO_0>..<AUDIO_16383>` 解析成 id，从基模的输入嵌入表里**拷贝这 16384 行**，
  用来初始化 `vocal_embed`、`acc_embed`、`acc_head`，从而继承基模学到的音频先验。
  解析失败时：`warm_start_strict=True` 直接报错（推荐首次训练打开），否则退化为随机初始化。
- 拷贝完后**删除基模庞大的 `embed_tokens` 和 `lm_head`**（约 1.7 亿参数）以省显存——
  因为我们不走原生 token 嵌入路径，而是用自己的双轨嵌入。

**(2) 新增的可训练参数**
```python
vocal_embed = nn.Embedding(16385, E, padding_idx=16384)   # 人声轨嵌入
acc_embed   = nn.Embedding(16386, E, padding_idx=16385)   # 伴奏轨嵌入（含 MASK 行）
input_proj  = nn.Linear(2E, D)                            # 逐帧拼接后投影到骨干维度
acc_head    = nn.Linear(D, 16384, bias=False)            # 伴奏预测头
rtd_head    = nn.Linear(D, 1)                            # （可选）RTD 判别头
```

**(3) 逐帧双轨嵌入 `embed_input`**
```python
ev = vocal_embed(vocal_ids)            # (B,T,E)
ea = acc_embed(masked_acc)             # (B,T,E)
x  = input_proj(cat([ev, ea], dim=-1)) # (B,T,D)  ← 论文 Eq.1-2
```

**(4) 双向骨干前向 `_run_backbone`（去因果掩码，本项目的关键改造）**
- Qwen3 默认是因果注意力。这里**绕过** `Qwen3Model.forward` 自动生成的因果掩码，
  直接逐层调用 `backbone.layers`，并传入一个我们自己构造的**双向加性注意力掩码**：
  形状 `(B,1,L,L)`，对 padding 的 key 位置填 `finfo.min`，其余为 0（不带因果三角）。
- RoPE 位置编码用 `backbone.rotary_emb` 在 `position_ids = 0..L-1` 上计算。
- 这样每个伴奏帧都能「看到」整段人声和整段伴奏，实现全序列双向建模。
- （已在自测中验证：改动最后一帧会影响第一帧的 hidden state，证明确实是双向。）

**(5) 训练前向 `forward`**
1. `mask_sampler.sample` 采样每个样本的 `t`；
2. `forward_mask` 生成 `masked_acc` 和 `mask_positions`；
3. 双轨嵌入 → 双向骨干 → `acc_head` 得到 `logits (B,T,16384)`；
4. `_cml_loss`：只在被掩码位置算交叉熵，按样本求 `Σce /(t·T)` 再对 batch 平均
   （`ignore_index=ACC_PAD_ID` 自动忽略 padding）；
5. 若开启 RTD：`_rtd_loss` 做第二次前向（见下），`loss = cml + 0.2·rtd`。

**(6) RTD 损失 `_rtd_loss`（ELECTRA 风格）**
- 在被掩码位置用模型自己的 argmax 预测（**detach**）填充，得到「被污染序列」`corrupt`；
- 标签 `d_i=1` 表示该位是原始 token，`d_i=0` 表示被替换（预测≠真值）；
- 用同一个骨干对 `corrupt` 再前向一次，`rtd_head` 输出每位「是否原始」的概率，做 BCE。
- 代价：多一次前向。可用 `model.use_rtd=false` 关闭以提速。

**(7) 推理接口**
- `predict_logits`：单次前向得到伴奏 logits（解码器调用）。
- `predict_logits_cfg`：仅当启用文本条件时做 classifier-free guidance（本项目默认不用）。

### 3.6 `lada_band/data.py` —— 数据读取与解析
支持两种数据格式（由 `data.format` 切换）：

**(A) `messages` 格式（你的数据，默认）**
- 正则解析：从 user turn 的 `[VOCAL_SOA]..[VOCAL_EOA]` 区间提取人声 `<AUDIO_x>`，
  从 assistant turn 的 `[SOA]..[EOA]` 区间提取伴奏 `<AUDIO_x>`；`<AUDIO_x>→整数 x`。
- **多轮自动拼接**：若一首歌按段落分多轮，各区间按消息顺序拼接成完整序列。
- **文本 prompt 被忽略**（即你选择的纯人声方案）。
- `[SOA]` 不会误匹配 `[VOCAL_SOA]` 内部。
- `messages_to_pair` / `messages_to_vocal` 分别取（人声,伴奏）/ 仅人声。
- `ids_to_audio_str`：反向把 id 序列包成 `[SOA]<AUDIO_x>..[EOA]`（推理输出 jsonl 用）。

**(B) `pt` 格式（备选）**
- manifest 每行 `{"vocal":"...pt","acc":"...pt"}`，用 `load_codec_pt` 读取。

**关键工程点：大文件不进内存**
- `JsonlLineIndex`：初始化时只扫描一遍记录**每行的字节偏移**，`__getitem__` 时再
  `seek` 到该行解析。10 万首歌的巨型 jsonl 也只占很小内存（偏移表）。
- `V2ADataset`：负责随机裁剪到 `max_frames`、过滤过短样本、对齐人声/伴奏到较短长度。
- `V2ACollator`：把一个 batch 右 padding 到最长，生成 `vocal_ids/acc_ids/pad_mask`。

### 3.7 `lada_band/generate.py` —— 迭代解码（**MaskGIT 式 confidence-based 并行解码**）
这就是「一次性采所有位置 → 保留置信度最高的一批 → 其余 remask 再采 → 迭代收敛」的标准
MaskGIT 解码。`generate_accompaniment(model, vocal_ids, pad_mask, num_steps, temperature,
top_k, top_p, mask_temp, remask_strategy)`：
1. 伴奏初始化为全 `[MASK]`（padding 处保持 pad）；
2. 每步**一次并行前向** `predict_logits`，对**所有位置**用 `sample_tokens` 采样并得到置信度；
3. 余弦调度算这一步结束后要保留的掩码数 `n_mask = floor(cos(π/2·ratio)·length)`，
   随 `ratio=step/num_steps` 从 1→0 单调下降，保证收敛；
4. 置信度分数 = `log(prob) + mask_temp·(1-ratio)·Gumbel噪声`（mask 温度，早期扰动大、后期趋稳）；
5. 取置信度**最低**的 `n_mask` 个位置重新掩码，其余（置信度高的）保留进入下一步；
6. 末步 `n_mask=0`，全部揭示；残留掩码用 argmax 兜底。
- `split_to_list`：把 padding 的 `(B,T)` 还原成每条样本的真实长度序列。

**两种 remask 策略（`remask_strategy` 参数）：**
- `"progressive"`（默认，经典 MaskGIT / LaDA-Band）：token 一旦被揭示就**锁定**，
  只有当前掩码位参与重掩码竞争（已揭示位置 `+inf` 保护）。掩码集合单调收缩。
- `"reconsider"`：每步对**所有有效位置**（含已揭示的）按当前置信度重新排序，
  早期置信度低的已揭示 token 也可能被重新掩码、后面再采。对早期错误更鲁棒，略慢一点点。

### 3.8 `scripts/train.py` —— 训练主循环
- **分布式**：读 `torchrun` 注入的环境变量初始化 DDP（单卡/CPU 也能跑）。
- **配置覆盖**：`--section.key value` 可覆盖 YAML 任意字段（如 `--train.learning_rate 2e-5`）。
- **优化**：AdamW（lr 1e-5、wd 0.01）、**余弦学习率 + 10k warmup**、梯度裁剪 1.0、
  **bf16 autocast**、梯度累积（`no_sync` 在累积期间跳过 DDP 通信提速）。
- **检查点**：按 `save_every` 存 `model.pt + trainer.pt + config.yaml`，并按
  `save_total_limit` 自动清理旧的；支持 `resume_from` 续训（恢复权重/优化器/调度器/步数）。

### 3.9 `scripts/infer.py` —— 推理
- 读 `messages` 或 `pt` 格式的测试集，取出人声 token；
- `generate_one`：整曲生成（默认）或可选分块（`--chunk_frames`/`--chunk_overlap`，给超长曲省显存）；
- 输出保存为 `[1,1,T]` 的 `.pt`，**可直接喂给 Muse 的 `infer/decode_audio.py` 解码成音频**；
- `--out_jsonl`：额外输出 `[SOA]<AUDIO_x>..[EOA]` 格式的 jsonl，方便接你的评测流程。

### 3.10 `scripts/build_manifest.py`（仅 pt 格式需要）
按文件名 stem 把 `vocal_dir` 和 `acc_dir` 里的 `.pt` 配对成 manifest，可选 `--check_lengths`
预先剔除长度严重不一致的样本。**用 messages 格式时不需要这一步**。

### 3.11 `scripts/smoke_test.py` —— 自测
不需要真模型/真数据，用一个超小随机 Qwen3 在 CPU 上验证：数据解析、双向性、
CML/RTD/条件三种训练路径、迭代解码。随时 `python scripts/smoke_test.py` 自检。

## 四、数据格式

### 4.1 messages 格式（推荐，你的数据）
每行一个 JSON：
```json
{"messages": [
  {"role": "user",      "content": "Pop, 女声, ...[VOCAL_SOA]<AUDIO_12><AUDIO_77>...[VOCAL_EOA]"},
  {"role": "assistant", "content": "[SOA]<AUDIO_5><AUDIO_91>...[EOA]"}
]}
```
- 人声取自 user 的 `[VOCAL_SOA]..[VOCAL_EOA]`，伴奏取自 assistant 的 `[SOA]..[EOA]`；
- `<AUDIO_x>` 的 `x` 即 0–16383 的 MuCodec 码本下标；
- `[VOCAL_SOA]` 前的文本会被忽略；多轮会按顺序拼接。

### 4.2 pt 格式（备选）
```json
{"vocal": "/data/vocal/song1.pt", "acc": "/data/acc/song1.pt", "name": "song1"}
```
每个 `.pt` 是 MuCodec `sound2code` 的输出（`[1,1,T]`）。

## 五、具体使用步骤

### 步骤 0：安装环境
```bash
pip install -r requirements.txt     # torch、transformers==4.57.3、pyyaml、numpy、tqdm
```
> 音频↔token 的编码/解码用 Muse 的另一套环境（`Muse/requirements_mucodec.txt` +
> `Muse/train/encode_audio.py` / `Muse/infer/decode_audio.py`）。

### 步骤 1：准备基模
把 `configs/base_0.6b.yaml` 里的 `model.pretrained_backbone` 改成**你的扩展词表基模路径**
（或训练时用 `--model.pretrained_backbone /path` 覆盖）。基模需要包含 `<AUDIO_0..16383>`
（用于 warm-start）。验证 token 命名：
```bash
python -c "from transformers import AutoTokenizer; \
t=AutoTokenizer.from_pretrained('/path/to/你的基模'); \
print(t.convert_tokens_to_ids('<AUDIO_0>'), t.convert_tokens_to_ids('<AUDIO_16383>'))"
```
返回两个有效（非 unk）的 id 即可。若命名不同，设 `model.audio_token_template`。

### 步骤 2：准备数据
messages 格式无需转换，直接把 `data.train_manifest` 指向你的 jsonl 即可。

### 步骤 3：训练
单卡：
```bash
python scripts/train.py --config configs/base_0.6b.yaml \
    --model.pretrained_backbone /path/to/你的基模 \
    --model.warm_start_strict true \
    --data.train_manifest /path/to/train.jsonl
```
8 卡：
```bash
NGPU=8 bash train.sh \
    --model.pretrained_backbone /path/to/你的基模 \
    --data.train_manifest /path/to/train.jsonl
```
**两阶段课程（论文 4.4 节）**：先短片段（`max_frames=1024`，约 40.96s）训练，
再用整曲（`max_frames=4096`）微调：
```bash
NGPU=8 bash train.sh \
    --data.max_frames 4096 --train.max_steps 40000 \
    --train.resume_from runs/lada_band_0.6b/checkpoint-150000
```
训练日志里出现这行即说明 warm-start 成功：
```
[lada_band] warm-start: resolved 16384 audio rows, token-id range [.., ..] ...
[lada_band] warm-started vocal/acc embeddings + acc head from <AUDIO_*>.
```

    --ckpt runs/lada_band_0.6b/checkpoint-150000 \

## 七、关键实现细节与常见问题 (FAQ)

**Q1：双向注意力是怎么实现的？为什么不会变回因果？**
A：我们不调用 `Qwen3Model.forward`（它会自动加因果掩码），而是逐层调用 `backbone.layers`，
传入自己构造的 `(B,1,L,L)` 加性掩码——只屏蔽 padding，不带因果三角。Qwen3 的注意力把
掩码**加**到注意力分数上；只要掩码非 None，SDPA 内核就不会启用 `is_causal`，从而实现双向。

**Q2：基模路径在哪指定？**
A：在配置的 `model.pretrained_backbone`，或命令行 `--model.pretrained_backbone /path` 覆盖。
不是命令行必填项，所以示例命令里没显式出现。

**Q3：`[VOCAL_SOA]` 这些标记会喂给模型吗？**
A：不会。在双轨设计里它们只是**解析分隔符**，模型吃的是两路对齐的 `<AUDIO_x>` 整数 id
（分别过 `vocal_embed`/`acc_embed`），不是扁平 token 流。你扩展词表里的这几个标记只用于
warm-start 不到（它只需要 `<AUDIO_*>`），其余不影响。

**Q4：怎么确认 warm-start 真的生效了？**
A：看训练日志是否打印 `warm-start: resolved 16384 audio rows ...`。首训建议
`--model.warm_start_strict true`，解析不到会直接报错，避免悄悄退化成随机初始化白训。

**Q5：显存不够 / 序列太长怎么办？**
A：注意力是 O(L²)。可以：① 减小 `data.max_frames`；② 推理时用 `--chunk_frames` 分块；
③ 关闭 RTD（`--model.use_rtd false`）省一次前向；④ 用更小 batch + 更多梯度累积。

**Q6：推理用的是什么解码算法？步数怎么选？**
A：用的是 **MaskGIT 式 confidence-based 并行解码**（见 [3.7](#37-lada_bandgeneratepy--迭代解码maskgit-式-confidence-based-并行解码)）：
每步一次并行前向采所有位置，保留置信度最高的一批、其余按余弦调度重新掩码，迭代收敛。
`--num_steps` 20 是质量/速度折中（最多可到 60）；`--remask_strategy` 选 `progressive`
（默认，揭示即锁定）或 `reconsider`（已揭示低置信也可回炉）；`--mask_temp` 调重掩码随机性。

**Q7：人声和伴奏长度不一致？**
A：V2A 假设两者逐帧对齐，数据集会自动截断到较短者。若你的数据存在系统性偏移，
建议在数据侧先对齐。

**Q8：能加回文本/风格条件吗？**
A：可以。`model.use_condition_prefix=true` 会启用一个全局条件前缀 token + classifier-free
guidance（需要你额外提供条件向量）。当前按你的要求默认关闭、纯人声。

**Q9：为什么不用 ms-swift 训练？**
A：ms-swift 面向因果 SFT，无法表达「双向注意力 + 掩码扩散 + RTD」这套 NAR 目标，
所以这里是独立的 `torchrun` DDP 训练循环。
